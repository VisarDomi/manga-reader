04 Bloat Candidates

Ranked by confidence. "Bloat" here means complexity that may not own enough value to justify its code/data/runtime cost.

Index Previous: Edge Cases Next: Options

Candidate Ranking

RankCandidateEvidenceLikely simplificationRisk
1 All-eager chapter page-map metadata jobs Comix has ~2.86M queued cache-chapter-page-map jobs; Mangadot has ~185k. This is the largest non-source bloat source. Replace "cache everything eventually" with "cache newest/favorites/search-observed plus reader-near-window demand." Cold first-open cost can return for obscure entries unless foreground priority is excellent.
2 CacheService.ts as mega-owner 1,940 LOC, 77 timer/async hits, owns jobs, filter catalog, store policy, foreground waits, data parsing, covers. Split by resource: MangaDataCache, ChapterImageMetadataCache, FilterCatalogCache, StoreSelectionPolicy, sharing one durable queue. Wrong split can add code. The split must move policy out, not wrap existing methods.
3 ReaderState + Reader.svelte coupled ownership 3,182 LOC total. Model and DOM both participate in projection, rebase, measurements, image scheduling, diagnostics. Define one ReaderProjectionOwner contract: state computes projection, component only applies one returned DOM command. Reader is fragile. Rewrite in tiny proofed steps, not broad replacement.
4 Provider behavior outside provider capability layer Provider sources are only 806 LOC while runtime fallbacks and byte policy live in generic server services. Move provider-specific runtime strategy to typed provider capabilities consumed by generic services. Can become over-abstracted if only two providers exist. Keep capability surface small.
5 Frontend bundled provider JS app/src/lib/services/bundled/comix.js and mangadotnet.js total 885 LOC. Verify usage; if stale, delete the files and any path that imports provider parsing on frontend. May be used by old client-side API parsing. Must prove with rg before deleting.
6 Restore logic embedded in app-wide state hub AppState owns restore shell, root replay, provider filters, search pagination, session persistence. Extract RestoreCoordinator with typed layer-prepare commands. AppState wires domains but does not implement restore policy. May add files but reduce total policy complexity. Success metric should be fewer restore branches in AppState.
7 Cover/image diagnostics distributed across UI and services Current logs are useful, but cover summaries dominate the window. Centralize diagnostic level and aggregate windows; keep one summary per transition or route. Too much compaction can hide rare image failures. Keep enough fields to reconstruct owner/source.

What Not To Delete Blindly

Reader bounded scroller

It looks complex because the edge case is complex. The simplification target is ownership, not removing the physical window.

BrowserSession warm runtime

The browser cost is real, but it replaced worse cold behavior and Cloudflare/signed API failures. Keep, but narrow provider policies.

Restore backing roots

This fixed real swipe-back failures. Do not return to lazy hidden roots; instead make root preparation a smaller explicit contract.

Structured logs

The logs are a development tool for this project. Remove spam, not evidence. Missing logs repeatedly caused wasted test opportunities.

Likely Shoe-Horned Edge Cases

BehaviorWhy suspectDecision prompt
Reader recommendations tail behaves like manga-detail recommendations. It forced reader geometry/navigation to understand recommendation cards and return targets. Is it worth keeping inside reader, or should "end of latest chapter" be a simple button/portal to manga detail recommendations?
Daily all-provider all-search crawl. It is provider-expensive and fills queues massively. Does the app need whole-catalog freshness, or only newest/search/favorites freshness?
Multiple retry/wait layers around cache foreground requests. Some are needed, but several exist because foreground/background priority was historically weak. After durable queue priority is trusted, can foreground waiters be reduced to "serve cache, subscribe to resource-ready"?
Frontend provider bundles. Likely survived from old architecture where frontend parsed provider results. Can the frontend be pure consumer of backend/provider-shaped API?
