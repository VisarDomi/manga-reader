05 Restructure Options

Options for reducing code and conceptual load without dropping the core edge cases. These are not implementation plans yet; they are decision surfaces.

Index Previous: Bloat Candidates

Option A: Resource-Oriented Cache Split

RECOMMENDED Split CacheService by resource ownership while keeping one durable queue implementation.
DurableQueueclaim/retry/reap/yield only
→
MangaDataCachemanga details, chapters, recommendations, filter harvest
→
ImageMetadataCachechapter page maps, store candidates, scrambled metadata
→
ByteCachecovers/thumbnails/provider bytes
Expected gainRiskAcceptance test
Lower cognitive load. Policies move from one 1,940 LOC file into resources that can be reasoned about independently. Can add files without reducing complexity if methods are only moved mechanically. Each cache resource answers: what do I serve, when do I refresh, what jobs do I enqueue, what logs prove it?

Option B: Change Background Data Policy

PRODUCT DECISION Keep cache-first UX, but stop pretending whole-catalog chapter image metadata is always worth eager background work.
PolicyCode/data effectUser-visible effect
Current: newest crawl eventually queues every manga/chapter image map. Millions of jobs, GB DB, long-running background service. Potentially warm obscure manga, but costly and often not useful today.
Lean: cache newest search pages, favorites, opened manga, and reader-near-window images. Fewer jobs and smaller DB. More foreground misses for obscure first opens. Likely same day-to-day UX if foreground priority is strong.
Hybrid: whole-catalog manga/chapter list, on-demand chapter image maps. Keeps discovery metadata while removing largest job family. Opening a never-read chapter may wait once, but manga detail stays rich.

Option C: Reader Projection Contract

HIGH VALUE Do not rewrite the reader. Instead, enforce one contract: ReaderState computes the logical/physical projection and returns at most one DOM scroll command; Reader.svelte applies it and reports measurements.
DOM eventscrollTop, viewport, visible/measurements
→
ReaderProjectionslots, frame epoch, image warm candidates, optional scroll write
→
Componentapply scroll write, mount slots, schedule images
→
Logsone event explaining projection decision

This preserves the bounded physical scroller and iOS momentum constraints. It reduces bloat by making the DOM side less intelligent.

Option D: Provider Capability Objects

GOOD WITH 2+ PROVIDERS Keep generic cache/route/browser services, but stop encoding provider choices as scattered provider.id === ... checks.
CapabilityCurrent symptomBetter owner
Search transportSearch route chooses proxy/document/runtime fallback.Provider declares search transport strategy.
Image byte fetchByte cache receives provider-specific runtime byte callback.Provider image capability exposes fetchByte if required.
Scramble detectionCache/browser/decoder share knowledge.Provider image capability says whether page needs decode and how.
Filter catalogCache service fetches/harvests provider filters.Provider catalog capability returns typed filter catalog; cache only stores it.

Option E: Restore Coordinator Extraction

MEDIUM VALUE This may reduce AppState complexity, but only if it removes branches from AppState. A wrapper that calls the same methods in the same order is bloat.

The target shape is a typed restore plan: 

foreground: Reader | Manga | Search | Favorites | Providers | Comments
backingLayers: [Manga, Manga, SearchRoot | FavoritesRoot | ProvidersRoot]
for each layer:
  mount shell immediately
  ask owning feature to prepare hidden data
  feature applies own scroll when its data height is ready

Anti-Options

Tempting cleanupWhy it is bad
Delete reader virtual window and rely on normal DOM. Breaks the exact iOS Safari fast-scroll edge case the app exists to solve.
Disable hidden backing-root preparation. Reintroduces swipe-back blank/slow root bugs.
Remove logs because they are noisy. Historical fixes depended on logs. The right move is aggregate logs, not blind logs.
Make provider runtime cold to save memory. Cold runtime makes foreground UX depend on Cloudflare/browser startup. Use provider enable/disable instead.
Add a global "gate" for every race. Gates hide ownership bugs and add latency. Use explicit resource/session ownership or bounded rollback instead.

Suggested Cleanup Order

  1. Verify/remove frontend bundled provider code if unused.
  2. Decide background chapter image metadata policy. This can remove the most data bloat without touching reader UX.
  3. Split CacheService by resource ownership, measured by reducing policy branches in the original file.
  4. Tighten reader projection contract, one small proofed step at a time.
  5. Extract restore coordinator only if it deletes meaningful branches from AppState.
  6. Move provider-specific runtime choices into provider capabilities.
Best near-term bet: the fastest path to "less code, more beautiful architecture" is not a rewrite. It is deleting stale frontend provider bundles, reducing the all-eager chapter-image metadata policy, and splitting CacheService only where it removes real policy coupling.
