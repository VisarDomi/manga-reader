Top-Level Classification

AreaStatusWhy
Reader bounded physical window KEEP Necessary for iOS Safari and fast infinite scrolling. Complexity is real, but ownership is still concentrated in too many places.
Provider-specific browser runtime KEEP Necessary for Comix/Mangadot signed/runtime data and Cloudflare. Should remain provider-owned.
Durable eager chapter-image metadata cache QUESTION It handles offline-ish warm paths, but current queue volume suggests the policy may be larger than the value.
Restore/root preparation machinery KEEP Real edge case: swipe-back layers must already be usable. The current implementation is better, but still lives in the broad app state hub.
Duplicated bundled provider code in frontend REMOVE? Likely leftover from earlier architecture. Needs verification before deletion, but it is suspicious because provider parsing should be backend/provider-owned.

How To Read This Report

The report uses three labels: KEEP means the complexity maps to an edge case the app currently needs; QUESTION means the code may be valid but the product value should be decided; REMOVE? means the evidence points to likely dead or poorly owned code, but deletion still needs a code/test pass.
