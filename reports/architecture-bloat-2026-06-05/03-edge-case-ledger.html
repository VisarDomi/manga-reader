03 Edge Case Ledger

This page separates necessary edge-case handling from optional or questionable behavior. The goal is to make deletion risk visible.

Index Previous: Code Map Next: Bloat Candidates

Required Edge Cases

Edge caseWhy it is realCurrent ownerCleanup direction
iOS Safari fast infinite reader scroll Unbounded height and top-prepend/removal break momentum and can flash/black-screen. ReaderState, Reader.svelte, ReaderWindowManager, ReaderMemoryManager. KEEP Complexity is justified, but DOM scroll writes should stay in one explicit owner.
Reader restore and progress App can reopen on reader; user expects page/progress continuity without hijacks. AppState, ReaderState, IndexedDB progress/session. KEEP Persisted state is needed; reduce cross-writes between restore, visible-page tracking, and close.
Swipe/back layer restoration Search/favs, stacked manga details, reader, and comments are real layers in the interaction model. AppState, UIState, feature states. KEEP Current fix improved backing-root preparation. Long-term split restore planning from feature hydration.
Cache-first manga data Provider runtime can be slow/unreliable; cached data makes app usable and often faster than site. CacheService, SQLite, cache routes, frontend API state. KEEP Serve cache immediately. Question only the eagerness/volume of background metadata.
Provider runtime / Cloudflare / signed APIs Comix and Mangadot need browser-owned runtime access for some paths. ProviderCoordinator, BrowserSession, provider descriptors. KEEP Necessary. Move more provider-specific policy into provider capability objects.
Scrambled Comix pages Some image files are visually scrambled. Site renders them through runtime logic. ScrambledPageDecoder, BrowserSession, CacheService. KEEP Needed for correctness. Keep warm and critical-priority aware.
Provider-specific filters Search semantics differ by provider; tag catalogs should not be handwritten by user every time. Provider cache, providerFilters routes/state, filter UI. KEEP Cache daily. Keep UI provider-scoped.
Comments on manga/chapter User-facing feature, provider-specific identifiers and thread shapes. CommentsService, comments routes, frontend comments sections. KEEP Should be provider-owned. Avoid browser use unless provider requires it.

Questionable Or Product-Optional Edge Cases

Edge case / behaviorWhy it may be bloatEvidenceUser decision needed
Eagerly caching chapter image metadata for everything It creates millions of durable jobs and GB-scale DBs. The user mostly needs foreground and near-future warm paths. Comix queue has ~2.86M queued cache-chapter-page-map jobs; DB is 5.4GB. DECIDE Keep all-eager, cap by recency/favorites, or make it on-demand plus daily newest.
Search/favorites root preparation while hidden Needed for swipe-back UX, but hidden-root work can compete with foreground if not sequenced. Current logs show root prep works; cover rendering still costs up to seconds after mount. KEEP Keep behavior, simplify into a root-preparation owner.
Recommendations at reader tail Useful, but it ties reader geometry to manga-detail card rendering and navigation return targets. Reader.svelte calculates recommendation tail height and renders MangaList. DECIDE If kept, make it a terminal slot type owned by reader model, not ad hoc component layout.
Multiple providers warmed at once Provider runtimes are expensive. The user added provider enable/disable because Mangadot cost was high. Browser surface logs show warm browser runtime uses 1.2GB RSS with Comix alone active in current run. KEEP Keep enable/disable. Maybe default disabled for costly providers is product policy, not code hack.
Dense diagnostics always present in source Logs are essential, but diagnostic flags and probes can become runtime/code noise if not centralized. Reader.svelte, PerfDiagnostics, and cover summaries all own logging decisions. DECIDE Keep lean aggregate logs; move debug probes behind one logging policy.
