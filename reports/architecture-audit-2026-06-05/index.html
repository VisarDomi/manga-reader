Main conclusion: the app is not simply suffering from one bad reader bug. It has several good architectural ideas, but some owners are still too broad. Restart clears symptoms because browser/runtime/durable-job state is not fully self-healing. The next fixes should strengthen owners rather than add gates or timeouts.

Recommended Next Fixes

  1. Add a provider-independent durable queue reaper for stale running jobs.
  2. Introduce explicit provider runtime states and foreground/background/decode lane ownership.
  3. Extract a frontend ScrollOwner and ProgressOwner around existing reader commands.
  4. Make every foreground cache miss return typed stale/refreshing/failed state instead of silent waits.
  5. Compact provider state-transition logs so future investigations do not drown in repeated request failures.
