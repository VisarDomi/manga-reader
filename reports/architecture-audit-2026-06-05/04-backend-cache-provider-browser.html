The scheduler does have lease-based recovery: claimNextJob() can reclaim stale running work, and recoverRunningJobsForOwner(workerId) exists on service start. The observed stale jobs show the recovery contract is incomplete.

Inference: recovery is tied too closely to the active worker path. If a provider is disabled, suspended, blocked by kind filters, or not claiming the same class of work, stale running jobs can remain visible in the DB long after they stopped doing useful work.

Runtime Browser Shape

ComponentWhat it owns todaySoft point
BrowserSession Warm runtime pages, foreground/background lanes, fetch through page runtime, provider page parsing, runtime health, context reset, decoder integration. One class owns too many browser lifecycles and policies.
ScrambledPageDecoder Warm decode page and screenshot-based descrambling for Comix images that require site rendering. Correctly separate from frontend, but still shares browser resource pressure with cache work.
CacheService Foreground stale/cache responses, background crawling, priority jobs, provider update policy, store policy, resource waiters. Foreground and background policy live together; user intent can wait on cache-owner internals.
ByteCacheService Thumbnail/cover bytes and image response ownership. Good boundary, but byte ownership must stay provider-local and never scan globally for a specific card request.

Failure Families From Cache DB

FamilyObserved examplesArchitecture interpretation
Runtime module drift Comix main module not found, runtime HTTP client export mismatch. Provider runtime discovery is brittle enough that logs should treat it as provider-degraded, not generic request failure.
Provider HTTP failures Large Comix 404/5xx families, Mangadot 500/502/521 and challenge cases. Provider availability and invalid manga/chapter data must be separate states.
Browser lifecycle failures context closed, browser closed, newPage failures, target closed. A browser/runtime owner can die while durable jobs still believe they are owned.
Incomplete page-map data Incomplete chapter images and empty page maps. Cache rows need explicit completeness states so the frontend does not confuse "known empty" with "still warming" or "bad cache".

Backend Restructure Direction

The best architectural fix is not another timeout. It is to make provider runtime, foreground requests, and background crawling explicit owners with bounded authority.

  • ProviderRuntimeActor: one per provider, with explicit disabled, warming, ready, challenged, degraded, and stopping states.
  • RuntimeLanePool: separate foreground, background, and decode lanes so background page-map work cannot poison visible image restore or comments/search work.
  • DurableQueueReaper: a provider-independent lease reclaimer that periodically returns stale running jobs to retry/queued and logs the action.
  • ForegroundRequestOwner: serve the freshest valid cache immediately, enqueue/subscribe to refresh, and never block visible UI on background queue depth.
  • BackgroundCrawlerOwner: owns daily crawl and eager cache, but can be suspended or degraded without changing foreground cache semantics.
  • Completeness contracts: cache entries should distinguish ready, stale, refreshing, incomplete, provider-failed, and not-found.
