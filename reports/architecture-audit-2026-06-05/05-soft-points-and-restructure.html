Anti-Pattern Matrix

PatternWhere it appearsWhy it mattersBetter owner
Timer as truth Reader recovery/progress/deep-sleep timers, cache waits and drain sleeps. Timers can be necessary, but they become bugs when they decide ownership instead of observing state. State-machine transition with a logged reason.
Catch and continue Cache foreground waiters and several browser/cache paths. Hides whether work was cancelled, failed, timed out, or superseded. Typed result: ready, refreshing, failed, aborted.
Large owner class reader.svelte.ts, CacheService.ts, BrowserSession.ts, AppState. Hard to prove one writer per resource because ownership is mixed inside large classes. Small command owners with narrow inputs and explicit outputs.
Foreground/background sharing without arbitration Provider runtime/browser/cache work. A background crawler failure can consume the same runtime health and browser resources as a visible user request. Separate lanes and a provider runtime actor.
Persisted state as implicit truth Session restore, favorite card snapshots, cache job rows. Old state can be structurally valid but wrong for the current provider or runtime. Normalize persisted state before replay, and version cache contracts.

Foreground waits while cache engine is busy. CacheService both serves foreground and consumes background queue. ForegroundRequestOwner serves stale cache immediately and subscribes to refresh completion.
Provider-specific breakage spreads into app behavior. Provider runtime failures are sometimes logged as generic request failures. Provider state machine and provider-scoped degraded/challenged states.

What Looks Worth Keeping

  • Bounded reader geometry: it directly addresses iOS/Safari scroll-height and momentum problems.
  • ProviderCoordinator: per-provider owners are the right direction; the next step is making runtime state explicit.
  • Durable SQLite jobs: the DB queue gives power-off recovery potential; it needs stronger stale-running reaping and compact status logs.
  • Byte/cache split: thumbnails/covers and provider data should remain separately owned.
  • Log-driven workflow: the app has enough event structure to investigate; the next improvement is decision-level summaries.

What Looks Like Bloat Or Drift

AreaWhy it is suspectSuggested action
cache_meta size DB stats showed roughly 172MB, unusually high for metadata. Audit keys and payload sizes; move noisy metrics to compact rollups or prune old data.
Runtime failure persistence Historical failures include stale module/runtime errors after provider behavior changed. Classify errors by provider runtime generation and expire stale runtime-drift failures.
Foreground waits with swallowed timeout FOREGROUND_CACHE_WAIT_MS plus ignored waiter errors can hide why UI waited. Return typed stale+refreshing state to frontend; log wait result explicitly.
Large owner files Several core files are near or above 1,000 lines. Extract command owners only when a boundary is proven by bugs, not as cosmetic refactor.

Recommended Path

  1. Add a DurableQueueReaper first. This is the smallest high-confidence backend fix because stale running jobs are observed facts.
  2. Split provider runtime state into a ProviderRuntimeActor. This should not rewrite provider parsing; it should make warm/degraded/challenged/stopping explicit and logged.
  3. Extract frontend ScrollOwner and ProgressOwner only around existing commands. Do not change reader geometry while doing this.
  4. Convert foreground cache misses to stale-now, refresh-async typed responses everywhere that user-visible UI can accept stale data.
  5. After those are stable, decide whether the cache/background engine needs a broader actor rewrite. Do not start there; too many unrelated edge cases would move at once.
