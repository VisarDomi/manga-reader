# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

### Changed
- **2026-02-25** — Strip all `[COMIX:*]` diagnostic logging from the codebase. Removed the `DEBUG` flag, `log` helper, `debugDump()`, `__comixDebug` window binding, the click interceptor in init, the `diag()` function in sentinel, `_snap()` in SearchState, and all `log.info/warn/error` calls across api.ts, search.svelte.ts, manga.svelte.ts, index.svelte.ts, sentinel.ts, and MangaCoverCard.svelte. Operational `console.error`/`console.warn` in catch blocks and the iOS resume sentinel remain. *Decision: full removal over keeping behind a dead-code-eliminated flag — the logging was added for a specific iOS debugging campaign that's resolved. Keeping it adds ~100 lines of noise that obscures the actual logic. If remote debugging is needed again, targeted `console.log` calls are easy to add and remove.*

### Fixed
- **2026-02-25** — Fix reader chapter navigation breaking on decimal chapters (3.1), gaps (10→12), and chapter 0. The old `getAdjacentChapterId` used `current.number + 1` arithmetic to find the next chapter — only worked for integer-spaced sequences. Replaced with index-based navigation on the filtered chapter list: reader stores the same sorted list from manga details and walks it by index. Also fixes reader loading chapters outside the selected group filter and ignoring group selection entirely. *Decision: index-based navigation on the filtered list over smarter arithmetic — the filtered list already represents exactly the chapters the user wants to read in order, so walking it by index is both simpler and correct for all edge cases.*
- **2026-02-25** — Deduplicate chapters with the same number when a group filter is active (keep latest upload per chapter number). When "All" is selected, every chapter from every group is shown — the user may want to pick between groups. *Decision: dedup only on group filter over always-dedup — "All" means show everything, dedup only makes sense when you've narrowed to one group and duplicate uploads are noise.*

### Changed
- **2026-02-25** — Remove titles from manga cards in search/favorites list — covers are self-explanatory, titles wasted space. Green chapter pill now shows for all manga with reading progress (e.g. `477/477` when caught up), not only when behind. Previously the green style only appeared when resume chapter differed from latest — a leftover from the old badge design that made caught-up manga look untracked. Buttons and filter chips bumped to 16px font-size to prevent iOS Safari double-tap zoom when tapping rapidly. *Decision: 16px font-size over viewport `maximum-scale=1` — the font-size threshold is iOS Safari's native heuristic for "readable text doesn't need zoom", preserving pinch-to-zoom for the reader while preventing accidental zoom on UI controls.*

### Fixed
- **2026-02-25** — Fix sentinel IO breaking after CSS visibility changes on iOS WebKit. When navigating list→manga→reader→list, iOS WebKit silently kills the IntersectionObserver after its root element goes through `visibility: hidden` transitions — the observer stays connected but never delivers callbacks. Added a `generation` parameter to the sentinel action; `UIState.setView()` increments `listViewGeneration` each time viewMode transitions to `'list'`, forcing the observer to disconnect and recreate. Also added a post-creation health check: after one rAF, if the sentinel is geometrically within the root+margin zone but the IO hasn't fired, calls `onIntersect()` directly as a fallback. *Decision: generation counter in UIState over `$effect` + `untrack()` in ListView — putting the counter at the source (where `setView()` is called) is idiomatic Svelte 5: imperative state updates in methods, no effects writing to state they read.*
- **2026-02-25** — Fix `each_key_duplicate` error when paginated search results contain the same manga across page boundaries. `loadNextPage()` now deduplicates by slug before concatenating. *Decision: dedup at the data layer over compound `{#each}` keys — preserves Svelte's keyed diffing optimization and prevents subtle DOM reuse bugs.*

### Added
- **2026-02-21** — Loading watchdog: if `isLoading` stays true for >15s, force-resets it, aborts in-flight requests, shows "Loading timed out — scroll to retry" toast, and forces a scroll container re-layout to recover from iOS WebKit touch handling desync (the `-webkit-overflow-scrolling: touch` bug where rapid DOM mutations inside a scroll container cause touch events to stop propagating). The re-layout toggles `overflow` off/on via `requestAnimationFrame`, which forces WebKit to recreate its internal scroll handler. `hasMore` stays true so the sentinel can retry on next scroll. *Decision: timeout watchdog over auditing every async code path — the bug has recurred across multiple "fixes" because the root cause is environmental (iOS WebKit scroll container desync), not a single code path. A safety net that catches any stuck state regardless of cause is more robust than patching individual race conditions.*
- **2026-02-21** — Show tags (genres, themes, demographics, formats) in manga detail view. Tags displayed as small chips below the author, resolved from `term_ids` in the comix.to API response against the local terms list. Unknown term IDs (comix.to-specific) are silently skipped. Term IDs also persisted through favorites. *Decision: flat `termIds` array on the Manga interface over separate `genres`/`themes`/`demographics` arrays — the API returns a single `term_ids` field and the local `TERMS` list already knows each term's category, so splitting at the type level adds complexity without benefit.*

### Changed
- **2026-02-21** — Refactor: moved global `styles.css` into scoped Svelte `<style>` blocks per component (SearchBar, FilterPanel, ListView, MangaList, MangaCoverCard, MangaView, ChapterList, Reader, Toast, +page.svelte). `styles.css` now only contains global resets (`*`, `button`, `html/body`) and the shared `.empty` utility. Also removed dead `.action-btn span` selector. *Decision: scoped component styles over single global stylesheet — colocates CSS with markup for better maintainability, leverages Svelte's built-in scoping, and eliminates unused selector warnings at build time.*
- **2026-02-21** — Manga card chapter display redesigned: resume and latest chapter shown inline with `/` divider. When they differ (user isn't caught up), both get the green pill styling. When same or no resume, just white text. Removed the old absolute-positioned green resume badge.

### Added
- **2026-02-19** — Favorites feature. Users can bookmark manga via a heart button in manga details and view all favorites from a "Favs" toggle in the search bar. Favorites stored locally on disk via the backend (`/home/visar/Pictures/manga-reader/favorites.json`). Backend: three endpoints (GET/PUT/DELETE `/api/favorites`). Frontend: `FavoritesState` with optimistic updates, Favs mode in ListView swaps search results for favorites list, heart button in MangaView header. *Decision: local JSON file over IndexedDB or upstream API — comix.to favorites are account-bound and we run in incognito mode, so local disk storage via the backend is the only reliable option. Optimistic UI updates with revert-on-error over waiting for the API response — keeps the heart toggle feeling instant.*

### Fixed
- **2026-02-19** — Fix reader only loading one next chapter at a time. The bottom sentinel fires `appendNextChapter()` which loaded exactly one chapter, but for short chapters the sentinel never re-triggered (IntersectionObserver only fires on transitions, and the sentinel stays within the 500% rootMargin the whole time). After each successful load, a one-shot IntersectionObserver now rechecks whether the sentinel is still within the 500% rootMargin zone — if so, it loads another chapter. No artificial chapter-count limit; the 500% rootMargin is the sole natural boundary. *Decision: one-shot IO recheck after each load over a loop with MAX_CHAPTER_DISTANCE cap — the viewport distance is the right limiting factor, not an arbitrary chapter count. The one-shot pattern naturally terminates when enough content pushes the sentinel out of the zone.*

### Added
- **2026-02-17** — Resilient pagination: (1) 12s frontend fetch timeout via `AbortSignal.timeout` + `AbortSignal.any` so fetches never hang forever. (2) Transient errors (timeout, network, 408/429/5xx) no longer permanently kill `hasMore` — sentinel stays active so next scroll retries. Only non-transient errors (404, parse) stop pagination. (3) `loadNextPage()` gets its own `AbortController` so `search()` abort doesn't leave `isLoading` stuck. (4) Pagination fetches now retry once with 1s backoff on transient failures. *Decision: `AbortSignal.any([callerSignal, timeoutSignal])` over replacing the caller's signal — preserves caller abort semantics while adding a hard timeout floor. Transient vs permanent error split avoids both infinite retry loops and premature death from a single flaky response.*
- **2026-02-17** — Diagnostic logging for load-more and manga-click bugs. All logs use `[COMIX:*]` prefix for easy filtering. Covers: sentinel lifecycle (`[COMIX:SENTINEL]` — create/intersect/update/destroy with DOM metrics), search pagination (`[COMIX:SEARCH]` — loadNextPage state snapshots, error categorization, abort detection), API calls (`[COMIX:API]` — URL, response shape, item counts), manga card clicks (`[COMIX:CLICK]` — DOM state, view-layer visibility/pointer-events), view transitions (`[COMIX:VIEW]` — openManga/closeManga with layer state). Also exposes `window.__comixDebug()` for on-demand state dump from console. *Decision: structured inline logging with state snapshots over a separate debug service — the goal is copy-paste-friendly console output that shows exactly what happened at each step, with enough DOM diagnostics to identify CSS/visibility issues.*

### Added
- **2026-02-16** — 3s debounce on filter changes. Toggling filter chips no longer fires a search per click — the search waits 3s after the last filter change, giving time to select/deselect multiple filters. *Decision: debounce in FilterState over SearchState — the debounce is a filter-interaction concern, not a search concern; search itself should remain instant when triggered by the user hitting the search button.*
- **2026-02-16** — Persist search filters and query across sessions. Filter state (term includes/excludes, types, statuses) saved to localStorage on every change and restored on startup. Search query also persisted. App init now searches with the last-used query+filters instead of empty. *Decision: localStorage over IndexedDB — filter state is a small JSON blob (~200 bytes), synchronous reads simplify constructor initialization, follows existing `contentMode` pattern.*

### Fixed
- **2026-02-16** — Fix load-more infinite retry loop. When `loadNextPage()` failed, `hasMore` stayed true so the sentinel immediately re-triggered the same failing page in a loop. Now sets `hasMore = false` on error.
- **2026-02-16** — Fix ListView sentinel using tiny 200px margin. Changed `SENTINEL_ROOT_MARGIN` to `500%` to match reader preload distance.
- **2026-02-16** — Fix reader showing unloaded images for a split second when scrolling. Two issues: (1) The IntersectionObserver's `rootMargin` was being ignored because `memory.root` was `null` when the observer was created (actions run during render, root is set in `$effect` after). Fix: `observePageImages` now accepts a `getRoot` getter and defers observer creation until root exists, with a pending queue flushed via `flushPageObserver()`. (2) Chapter sentinels used small fixed pixel margins (200px/500px) while image loading used a percentage-based margin — chapters weren't fetched until way after images would need them. Fix: all three observers (image loading, prepend sentinel, append sentinel) now use a unified `500% 0px` rootMargin, so next/prev chapters are fetched at the same distance images start loading. Also extended chapter reload range from ±1 to ±MAX_CHAPTER_DISTANCE. *Decision: unified 500% rootMargin over separate thresholds — ensures chapter fetch always happens before image loading needs the data, preventing the "loaded container but no images" gap.*

### Changed
- **2026-02-13** — Eight-part quality refactor across the codebase (no behavior changes):
  1. **Type safety** — Eliminated all `any` types in `api.ts` by adding `SearchApiResponse`, `ChaptersApiResponse`, `ChapterImagesApiResponse`, `HistoryApiResponse` interfaces to `types.ts`. Removed all 4 `eslint-disable` comments. *Decision: typed response interfaces over generic `Record<string, unknown>` — gives autocomplete and compile-time safety on response field access.*
  2. **Magic number extraction** — Created `src/lib/constants.ts` with named constants (`EDGE_ZONE`, `SWIPE_THRESHOLD`, `RESUME_RECOVERY_MS`, `DEEP_SLEEP_MS`, `VISIBLE_PAGE_RATIO`, `MAX_CHAPTER_DISTANCE`, `SCROLL_DEBOUNCE_MS`, `HISTORY_SYNC_MS`, `SENTINEL_ROOT_MARGIN`). Updated `swipeBack.ts`, `index.svelte.ts`, `ReaderMemoryManager.ts`, `ListView.svelte`, and new `PageTracker.ts` to import from constants. *Decision: single constants file over per-module constants — the values are cross-cutting concerns (e.g. `MAX_CHAPTER_DISTANCE` used by both memory manager and page tracker).*
  3. **Memoized derived state** — Fixed `ChapterList.svelte` `groups` from `$derived(() => ...)` (creates a getter returning a function) to `$derived.by(() => ...)` (evaluates and caches the result). Replaced `getEffectiveState()`/`getPrefix()` functions in `FilterPanel.svelte` with `$derived.by` maps (`effectiveStates`, `prefixes`) that recompute only when filter state changes. *Decision: `$derived.by` maps over per-render function calls — avoids recalculating every term's state on every render, especially with 50+ filter terms.*
  4. **Reader simplification** — Extracted scroll tracking and history sync debouncing into `PageTracker` class (`src/lib/services/PageTracker.ts`). `ReaderState` delegates to `pageTracker` for `trackVisiblePage`, `syncChapterProgress`, `flushVisiblePage`, and `clearHistorySync`. Reader.svelte delegates `handleScroll` to `pageTracker.handleScroll()`, removing `findVisiblePage` and `scrollTimer`. *Decision: dedicated `PageTracker` class over keeping logic in ReaderState — separates timer management (an imperative concern) from reactive state, and makes the reader component thinner.*
  5. **Standardized observer patterns** — Replaced ListView.svelte's 19-line inline IntersectionObserver with the existing `use:sentinel` action (3 lines). Added `disabled: isLoading` to prevent firing during load. Added shared-observer pattern documentation to `observePageImages.ts` and `observeChapterBoundary.ts`. *Decision: reuse existing sentinel action over keeping a one-off observer — identical behavior, consistent pattern across all infinite-scroll views.*
  6. **ProgressState O(1) update** — Changed `update()` from `this.all = { ...this.all, [slug]: data }` (O(n) spread creating a new object) to `this.all[slug] = data` (O(1) direct assignment). Svelte 5's `$state` proxy tracks granular property changes, so direct assignment triggers fine-grained reactivity. *Decision: direct assignment over spread — eliminates unnecessary object allocation on every scroll-triggered progress save.*
  7. **ConnectionMonitor cleanup** — Added `listen()` helper that stores cleanup functions, and `destroy()` method that removes all event listeners. Added `AppState.destroy()` that calls `monitor.destroy()`, `watchdog.stop()`, and `stopBackgroundSentinel()`. *Decision: cleanup array pattern over storing individual handler references — scales to any number of listeners without naming each one.*
  8. **IndexedDB error logging** — Added `console.error('[db] ...')` to the 3 silent `onerror` handlers in `db.ts` (`getProgress`, `setProgress`, `getAllProgress`). Errors still resolve gracefully (null/empty) but are now visible in devtools. *Decision: console.error + graceful resolve over throwing — IndexedDB failures shouldn't crash the app, but silent swallowing makes debugging impossible.*

### Fixed
- **2026-02-13** — Fix reader scroll position not persisting + auto-scroll to current chapter in manga details. The 3s debounced `syncChapterProgress` was cancelled by `clearHistorySync()` on reader teardown, so `pageIndex` was never saved to IndexedDB. Fix: track visible page in-memory via `trackVisiblePage()`, flush to DB immediately in `closeReader()` before clearing state. Also: manga details view now auto-scrolls chapter list to the last-read chapter on open via a `scrollIfCurrent` Svelte action, and resets scroll to top when loading a new manga. *Decision: in-memory `lastVisiblePage` flushed on close over reducing the debounce timer — separates the fast in-memory tracking (500ms scroll debounce) from the slow remote sync (3s chapter-change debounce), guaranteeing the page position is always saved even on immediate close.*
- **2026-02-13** — Fix prepend sentinel firing permanently and add scroll position restoration. The prepend sentinel used `root: null` (viewport) because `bind:this` runs after render, causing the sentinel to be permanently "intersecting" and fire only once. Sentinels now accept a `getRoot` function and a `disabled` flag — they stay disabled until scroll restoration completes, ensuring the root element exists when the IntersectionObserver is created. `observeChapterBoundary` similarly defers observer creation via `getRoot` and recreates when root transitions from null to element. Scroll position (page index) is now saved to IndexedDB on scroll and restored when reopening the same chapter. *Decision: deferred sentinel activation gated by `sentinelsReady` state over eagerly creating observers with a polled root — cleaner lifecycle, single `rAF` gate after scroll restoration guarantees root validity and prevents sentinels from firing during initial layout.*

### Changed
- **2026-02-13** — Five-part refactor of api, reader, localStorage, and search filter concerns:
  1. **api.ts `fetchJson()` wrapper** — All API functions now use a shared `fetchJson()` that checks `res.ok`, classifies errors into network/timeout/HTTP/parse via `ApiError`, and supports optional 1-retry with 1s backoff for transient failures (408, 429, 5xx). `fetchChapterImages` and `updateHistory` opt into retry. *Decision: module-private `fetchJson` with `ApiError` class over a generic fetch wrapper library — keeps error taxonomy co-located with the API layer, and `retry: true` opt-in avoids retrying searches (which would fight the AbortController cancel-on-new-search pattern).*
  2. **`use:sentinel` action** — Extracted duplicated append/prepend IntersectionObserver sentinel logic from `Reader.svelte` into `src/lib/actions/sentinel.ts`. Both sentinels now use `use:sentinel={{ root, rootMargin, onIntersect }}`. *Decision: params object with callbacks over two separate actions — the observer boilerplate was identical, only rootMargin and callback differed.*
  3. **Reader DOM access cleanup** — Replaced all `document.getElementById('view-reader')` with a `bind:this` ref passed via Svelte `setContext`/`getContext('readerRoot')`. Removed `ReaderPageElement.__readerData` DOM augmentation; page data now lives in `ReaderMemoryManager.pageDataMap` (`Map<HTMLElement, ReaderPageData>`). Observer actions register/unregister pages via `memory.registerPage()`/`memory.unregisterPage()`. Reader root ref also flows to `observeChapterBoundary` and `sentinel` actions. *Decision: context getter `() => HTMLElement | null` over a writable store — the ref is only needed synchronously at action init time, and a getter avoids subscribing to store updates.*
  4. **`storage.ts` utility** — Created `src/lib/services/storage.ts` with typed `getJson<T>(key, fallback)`, `setJson(key, value)`, `getString`, `setString`, `remove` helpers. Replaced scattered `typeof localStorage !== 'undefined'` guards in `SearchState` and raw `localStorage.getItem/setItem/removeItem` + `JSON.parse/stringify` in `ChapterList.svelte`. *Decision: thin wrapper with SSR guard over importing localStorage directly — centralizes the `typeof` check and JSON parse error handling in one place.*
  5. **`FilterState` class** — Extracted filter toggle logic (term include/exclude map cycling, type/status set toggling, content mode, `buildFilters()`, `clear()`) from `SearchState` into `src/lib/state/filter.svelte.ts`. `SearchState` owns a `FilterState` instance at `this.filters`, constructed with an `onChange` callback that triggers re-search. All UI consumers updated to `appState.searchState.filters.*`. *Decision: callback-based `onChange` over an event emitter or `$effect` — keeps FilterState framework-agnostic and avoids circular dependency between filter mutations and search execution.*
